** Introduction

# In this scenario, one promising approach is to make use of
# simulation to achieve scientific results.

# [HPC, Sustainability and Enviroment]
High-Performance Computing (HPC) infrastructures are critical for
scientific advancement. However, with increasing concerns regarding
environmental sustainability, HPC clusters and data centers are
recognized as major electricity consumers cite:IAE2024. Simulators aim
to address these issues by creating a faithful model of the real
system, allowing researchers to perform predictive analysis and
validate 'what-if' scenarios without the operational costs or
constraints of using the actual machine.

Using simulations not only lowers the electricity consumption, but
also targets the lack of reproducibility in HPC cite:repro2024,
ensuring a controlled and reproducible environment for experiments. In
this context, SimGrid is a free and open-source computational
framework capable of simulating several HPC scenarios with
reproducibility and versatility, allowing researchers to simulate
large platforms while maintaining accuracy cite:simgrid2025.


In this work, we present the development of a simulator for the
"Parque Computacional de Alto Desempenho" infrastructure present at
INF/UFRGS cite:LPPD through the SimGrid programmatic interface Simgrid
For You (S4U), focusing on the network topology. We show comparative
results between the platform created through simulation and the real
cluster, including compute time and energy consumption.

** Platform Creation and Calibration

The PCAD platform is composed of different homogeneous partitions, we
initially selected only the =cei=, =draco=, =tupi= e =poti= partitions from
PCAD infrastructure to simulate, with the three first having
six nodes and the last having only five. We choose these partitions as
they are the more used ones, aiming to first create a generic approach
that later can be used in the whole cluster. The methodology for
modeling was divided into programmatic platform generation and system
calibration, focusing on hosts computational power and the network
topology. The simulated environment was created programmatically using
the SimGrid for You (S4U) C++ API with SimGrid v4.1. This approach
allowed for the flexible definition of the cluster hierarchical
structure, effectively mirroring the physical architecture of PCAD.


Several different experiments were executed for the system
calibration. All experiments were run using Guix to ensure
reproducibility. The host calibration for the simulated partitions
consisted on the execution of High-Performance Linpack (HPL)
cite:HPL2003. For each partition this HPL was executed 10 times in two
different machines. The mean results in Gflops was taken as the time
for a node in the partition. Simple network tools such as =iperf3= and
=ping= were used to initially measure information such as network
bandwidth and latency. This initial information as latter used
together with official SimGrid material for calibrating the network
and MPI factors for each partition cite:calibration.

** Platform Validation

# , each application focusing in one
# point of the model such as computation, communication bandwidth and
# latency.

After calibrating the platform we used different proxy applications to
verify and validate the create model. Proxy apps are scale models of
real HPC applications, exhibiting the same computation and
communication patterns as the massive application that it stands
for. We chose three different proxy applications [fn::
https://framagit.org/simgrid/SMPI-proxy-apps]. Lulesh cite:lulesh,
that simulates shock hydrodynamics on unstructured meshes, widely used
to evaluate compute performance. MiniVite cite:minivite, that
implements distributed graph clustering, stressing the network through
highly irregular MPI communication patterns. CoMD cite:comd, that
simulates atomic forces, designed to stress system memory and regular
MPI communication.


We then created a experimental design using these applications. The
three applications were executed in all the target partitions (cei,
draco, poti and tupi). We varied the number of nodes used for the
execution between one, two or four. Also, for each application we
executed with different problems sizes, one small that takes
approximately 1 second to execute and one medium that take
approximately 30 seconds to execute, all executed with eight MPI
processes. Lastly, as SimGrid does not simulate threads parallelism,
we defined =OMP_NUM_THREADS= as 1 in all experiments. It gives us a
total of 480 unique configurations. We randomly executed each
configuration ten times to tackle variety. This experimental design
was applied to both the SimGrid simulation and the real-world cluster,
and the SimGrid simulation was execute in one machine of the Cei
partition. The RAPL tool was used the measure the energy consumed in
each experiment.

sized experiments (represented as a bigger circle) in function of the
number of nodes. Small circles represents our observations. The time
is given by the application as output at the end of the
computation. The black bars centered around the mean represents the
standard error with a CI of 99%. The figure shows the results faceted
by partition and applications, whit the colors identifying where the
execution takes place, in the real cluster (=real=) or in the SimGrid
